# Design: uv-based Environment Management for SciQLop

**Date:** 2026-03-05

## Problem

SciQLop currently uses `pip install --target` to install workspace dependencies into a local directory, then prepends it to `sys.path` at runtime. This approach:

- Reinstalls packages already available in the base environment
- Ignores site-packages, leading to duplication and bloat
- Runs installation at runtime (slow startup, flaky)
- Provides no isolation or reproducibility guarantees
- Plugin dependencies are declared in `plugin.json` but never auto-installed

## Architecture

Two-layer environment model with inheritance:

```
Base install (PyPI / AppImage / DMG / MSIX)
  SciQLop + PySide6 + core deps + uv binary
    └── Workspace venv (--system-site-packages)
         plugin extras + workspace-specific extras
```

The workspace venv inherits the base environment via `--system-site-packages`. Heavy packages (PySide6, Qt bindings, etc.) are never duplicated. The workspace venv only contains the delta: plugin dependencies and user-specified packages.

## Launcher

The `sciqlop` CLI entry point acts as a supervisor process running in the base environment. It never starts the Qt application directly — it prepares the workspace environment and spawns the Qt app as a subprocess.

### Launcher flow

```
sciqlop [--workspace <name|path>] [file.sciqlop] [file.sciqlop-archive]

1. Parse arguments
2. Resolve workspace:
   - --workspace <name> → ~/.local/share/sciqlop/workspaces/<name>/
   - file.sciqlop → read manifest, resolve or create workspace
   - file.sciqlop-archive → import archive, create workspace
   - no argument → use default workspace
3. If workspace dir doesn't exist, create it (dir + manifest + venv)
4. Compute effective dependencies:
   - Read enabled plugins from global settings
   - Apply workspace plugin overrides (add/remove)
   - Collect python_dependencies from each enabled plugin's plugin.json
   - Merge with workspace requires
5. Generate pyproject.toml in workspace dir
6. Run `uv sync` (or `uv sync --locked` for imported archives)
7. Launch Qt app using workspace venv Python
8. Wait for exit:
   - Exit 0 → quit
   - Exit 64 → restart same workspace
   - Exit 65 (or similar) + workspace identifier → sync & launch new workspace
```

### Base environment detection

The launcher detects its own `sys.prefix` to determine the base environment. This works naturally across all deployment scenarios:

- PyPI install (`pip install sciqlop` / `uv pip install sciqlop`)
- AppImage (embedded Python + uv binary at `/opt/uv/uv`)
- DMG / MSIX (bundled Python)

No dedicated base environment management is needed.

### Workspace switching

The Qt app's welcome panel shows available workspaces. When the user selects a different workspace:

1. The Qt app writes the target workspace identifier to a known location (temp file or env var)
2. The Qt app exits with a "switch workspace" exit code (e.g., 65)
3. The launcher reads the target, syncs the new workspace venv, and relaunches

## Workspace structure

```
~/.local/share/sciqlop/workspaces/<name>/
├── workspace.sciqlop       # manifest (source of truth)
├── pyproject.toml          # generated by launcher from manifest + plugins
├── uv.lock                 # lockfile (generated by uv)
├── .venv/                  # workspace venv (--system-site-packages)
├── notebooks/
└── scripts/
```

The `pyproject.toml` is generated — users never edit it. The `.sciqlop` manifest is the source of truth.

## Manifest format (`.sciqlop`)

```toml
[workspace]
name = "My Magnetosphere Study"
description = "Studying reconnection events in 2024"

[plugins]
add = ["some_extra_plugin"]
remove = ["experimental_collaboration"]

[dependencies]
requires = ["matplotlib>=3.8", "scipy", "hapiclient"]
```

### Plugin resolution

Effective plugin list = global defaults + workspace `add` - workspace `remove`.

The launcher reads each enabled plugin's `plugin.json`, collects their `python_dependencies`, and merges them into the generated `pyproject.toml` alongside workspace `requires`.

## Archive format (`.sciqlop-archive`)

A zip file with `.sciqlop-archive` extension containing:

- `workspace.sciqlop` — manifest
- `uv.lock` — pinned dependency versions for exact reproducibility
- `notebooks/` — Jupyter notebooks
- `scripts/` — user scripts
- Other workspace files

### Import flow

1. Extract archive to new workspace directory
2. Run `uv sync --locked` to reproduce the exact pinned environment
3. Launch SciQLop with the imported workspace

### Export flow

1. Copy manifest + lockfile + workspace files into a zip
2. Exclude `.venv/`, `pyproject.toml` (generated), and other transient files
3. Save as `.sciqlop-archive`

## File associations

Two file types to register with the OS:

| Extension | MIME type | Action |
|-----------|-----------|--------|
| `.sciqlop` | `application/x-sciqlop-workspace` | Open workspace (resolve path, sync, launch) |
| `.sciqlop-archive` | `application/x-sciqlop-archive` | Import workspace (extract, sync, launch) |

Registration mechanisms:
- Linux: freedesktop `.desktop` file + MIME XML
- macOS: Launch Services (Info.plist UTI declarations)
- Windows: MSIX manifest file type associations

## Generated pyproject.toml

The launcher generates this from the manifest + plugin metadata:

```toml
[project]
name = "sciqlop-workspace-my-magnetosphere-study"
version = "0.0.0"
requires-python = ">=3.10"

[project.dependencies]
# From enabled plugins (plugin.json python_dependencies)
speasy = ">=1.6.1"
tscat = ">=0.4.0"
tscat_gui = ">=0.6.0"
# From workspace manifest
matplotlib = ">=3.8"
scipy = "*"
hapiclient = "*"
```

Note: the exact format may need adjustment based on how uv handles `--system-site-packages` venvs with `pyproject.toml` projects. The key constraint is that base packages (PySide6, etc.) must not be re-resolved or reinstalled.

## Bundled app considerations

- AppImage already ships uv as a standalone binary (`/opt/uv/uv`)
- DMG and MSIX bundles will follow the same pattern
- Base environment is read-only in bundles — workspace venvs write only to the workspace directory
- `SCIQLOP_BUNDLED` env var remains for edge cases (e.g., blocking git-based dependencies)

## Migration

Existing workspaces using the old format (`workspace.json` + `dependencies/` dir) need migration:

1. Detect old-format workspaces (presence of `workspace.json` without `workspace.sciqlop`)
2. Convert `workspace.json` fields to `.sciqlop` manifest (name, description, dependencies list)
3. Remove old `dependencies/` directory
4. Create workspace venv and sync via uv

The old `pip_process.py` runtime installation and `sys.path` manipulation code is removed.

## What this design does NOT cover

- Notebook list management in workspace manifest (deferred)
- Custom `python_path` entries in workspace manifest (not portable, removed)
- Per-plugin isolated environments (rejected — too complex for little benefit)
- Launcher-managed base environment (rejected — bootstrap problem)
